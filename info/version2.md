# План доработки WireGuard TURN Android (Версия 2) — ОБНОВЛЕНО

## 1. Цель
Обновить Android-клиент для полной совместимости с `vk-turn-proxy` (v2). Это включает:
1.  **Session ID (UUID)**: Идентификация всех потоков одной сессии.
2.  **True Load Balancing**: Распределение исходящего трафика между всеми активными потоками (вместо текущей "sticky" логики), что соответствует поведению эталонного Go-клиента.

## 2. Анализ текущей реализации Android-клиента
В текущем `turn-client.go` реализован "Hub", который:
- Принимает пакеты от WireGuard на локальный UDP сокет.
- Выбирает **один** "sticky" поток (`activeStream`) и отправляет все пакеты в него.
- Переключается на другой поток только если текущий перестал быть `ready`.

**Проблема**: Это неоптимально при наличии 16 потоков, так как нагрузка идет на одно соединение, а остальные простаивают (используются только сервером для downstream).

## 3. План реализации

### Этап 1: Обновление нативного слоя (Go)
*   **Файл**: `tunnel/tools/libwg-go/turn-client.go`
*   **Генерация сессии**:
    *   В `wgTurnProxyStart` генерировать `sessionID := uuid.New()`.
*   **Handshake протокол**:
    *   В `stream.run` после DTLS handshake добавить:
      ```go
      if _, err := dtlsConn.Write(sessionID[:]); err != nil { return err }
      ```
*   **Балансировка (Round-Robin)**:
    *   Заменить `activeStream atomic.Pointer[stream]` на механизм выбора из всех `ready` потоков.
    *   В цикле обработки `lc.ReadFrom`:
      ```go
      // Найти все готовые потоки
      var readyStreams []*stream
      for _, s := range streams {
          if s.ready.Load() {
              readyStreams = append(readyStreams, s)
          }
      }
      if len(readyStreams) > 0 {
          // Выбрать следующий поток (Round-Robin)
          s := readyStreams[atomic.AddUint64(&counter, 1) % uint64(len(readyStreams))]
          // Отправить пакет в s.in
      }
      ```

### Этап 2: Улучшение живучести
*   Поскольку сервер теперь тоже балансирует трафик (downstream), клиент уже корректно его принимает (каждый `stream` пишет в общий `out: lc`).
*   Добавить логирование (через `turnLog`) при изменении количества активных потоков в пуле балансировки.

## 4. План тестирования
1.  **Проверка Session ID**: Убедиться по логам сервера, что все потоки успешно авторизовались под одним UUID.
2.  **Проверка балансировки**:
    *   Включить 16 потоков.
    *   Нагрузить канал трафиком.
    *   Проверить (через логи или метрики), что пакеты распределяются по разным `[STREAM %d]`.
3.  **Endpoint Stability**: Проверить на стороне WireGuard сервера, что `endpoint` не меняется при переключении потоков внутри сессии.
